## 一、背景

v3-005 做了「**換手後的復原**」(Stop 機械寫交接、SessionStart 自動注回)。但它沒做「**該換手了的警示**」——目前甲方靠感覺判斷 context 何時太滿。v3 紀律寫「context 壓在約 100k token 以下、clear 重啟勝過 compact」,但這條線在現場是**看不見**的。

本工單補上那條看得見的線:一個常駐 **statusLine** context-用量量表,context 進入換手區時柔性提醒甲方「告一段落再交接」。

**已查證的關鍵事實(動工前,非臆測)**:Claude Code 的 statusLine 透過 stdin 餵入 JSON,其中 `context_window` 物件暴露**即時 context window 佔用量**(`used_percentage` / `total_input_tokens` / `context_window_size` / `current_usage{...}` / `exceeds_200k_tokens`)。語意自 CC v2.1.132 起 = **當前佔用量**(非 session 累計)。**本機 CC 2.1.177**,欄位在、語意對。來源:官方 statusLine 文件 + 本地版本確認。

---

## 二、設計原則(不可違背)

1. **statusLine 是純被動渲染面**。只能畫一行字,**不能擋、不能注入、不能中斷**;Claude Code 也**無**「context 漲過 N token」的事件 hook。警示上限 = 那行字在過線時轉提示。本工單不嘗試任何中斷/阻擋機制。
2. **門檻語意 = 軟線,非懸崖**(甲方拍板)。100k 是「開始找出口、留餘裕」的線,不是剎車。長 context 品質衰退是漸進的;~100k 是保守新鮮度線,故意設早。過線=柔性提醒「告一段落再交接」,**語氣非命令式**(不用「該換手了!」催停)。**單層**,不做第二層「真吃緊」,不做連續漸變。
3. **絕對門檻**。以 **絕對 100k input-side token** 為界,**不**隨視窗大小(200k / 1M extended)縮放——忠於紀律寫死的「~100k」。
4. **全域 = 全域風險**。statusLine 掛 user-level `~/.claude/` 後,會在**每個專案每次刷新**都跑。因此**永不弄壞狀態列**是硬要求(fail-soft、exit 0、不噴錯、要快),且「翻全域開關」排最後一片、前置安全閘綠了才翻。
5. **甲方逃生閥授權的例外**。標準紀律是「不碰 user-level `~/.claude/`」;本工單因甲方明確要全域顯示而**經授權**動 user-level,但**僅限那一個 statusLine wiring 鍵**;實作 script + 測試全留 repo,v3-005 三 hook 仍留專案級。回滾=刪那一鍵。
6. **可移植、純 bash 3.2**。本機 `/bin/bash` 3.2.57:禁 `declare -A` / `${var^^}` / `mapfile`。jq 主路徑 + awk fallback。hook/script 內零 LLM、零網路。

## 三、交付項目

### D1. `statusline/context-gauge.sh`(量表 script,實作主體)
- 從 stdin 讀 statusLine JSON,解析 input-side token 佔用(jq 主路徑 + 窄目標 awk fallback)。
- 與絕對 100k 線比較:線下 → 正常顯示用量;線上 → 柔性換手區提示 + 變色。
- fail-soft:`context_window` 缺欄 / JSON 壞 / jq 缺席 → 降級、不弄壞狀態列、exit 0。

### D2. `~/.claude/settings.json` 的 statusLine wiring(唯一動 user-level 處)
- 一個 `statusLine` 鍵,指向 repo 的 `context-gauge.sh` 絕對路徑,`type: command`。
- INSTALL 記裝法 + 「刪那一鍵」一鍵回滾。

### D3. `tests/`(純 bash test runner,沿用 v3-005 模式)+ INSTALL 補章
- 門檻邊界、兩態渲染、fail-soft、jq/awk parity 的表測。

## 四、切片計畫(垂直切,逐片 red-green、逐片驗收 + mutation 查核)

- **Slice 1 — 門檻核心 end-to-end**
  解析建於「已確認與本機 CC 2.1.177 相符的官方 `context_window` 規格」,且**寬容**(多欄位 fallback):primary `.context_window.total_input_tokens`;fallback = `.context_window.current_usage.{input_tokens,cache_creation_input_tokens,cache_read_input_tokens}` 之和。真實 payload 欄位名於 **Slice 3 dogfood 實跑時對齊**(statusLine 設定開機才讀、session 中途無法熱載,故真驗放它真的會跑的時機)。
  切過:解析 input-side token 佔用(jq + awk fallback)→ 比絕對 100k → 兩態渲染。
  驗收表:邊界 99k / 100k / 101k;線上含「換手區/告一段落」字樣 + 變色碼;線下顯示用量、無提示;語氣非命令式(斷言**不含**催停字眼如「該換手了」「STOP」)。

- **Slice 2 — 全域安全閘(fail-soft + 速度)**
  `context_window` 缺欄 / JSON 壞 / 空輸入 / 非預期型別 / jq 缺席 → 永不弄壞狀態列(降級、exit 0、stderr 不噴錯)。jq vs awk parity 表測。**本片綠了才准翻全域開關。**

- **Slice 3 — 真實全域 wiring + INSTALL + dogfood**
  寫 `~/.claude/settings.json` statusLine 一鍵 → 在別專案實地觀察量表會動、過線會柔性提示 → INSTALL 記裝法 + 一鍵回滾。唯一動 user-level、影響所有專案的一步,壓最後。

- **加分項(排主線後,甲方決定做不做)**:量表同時顯示 model / %;接近線的預警層(甲方目前明確只要一層,預設不做)。

## 七、實機驗收後的轉折(2026-06-16,dogfood 結論)

statusLine 三片做完、13 test 綠、mutation 驗收過——**腳本本體沒問題**。但甲方重啟親驗時發現:

**核心發現**:甲方日常用的是 **VS Code 原生擴充的「CLAUDE CODE」側邊面板**,不是終端機 `claude` CLI。`statusLine` 是 CLI/TUI 的功能,**只渲染在終端機最底那行**,側邊面板有自己的 GUI、不渲染 command 型 statusLine。等於我們建的量表**打不到甲方真正在用的介面**——腳本沒壞、設定沒錯,是**呈現面選錯了**。

**GitHub 查證(甲方要求,非閉門造車)**:
- 社群做 context 量表清一色走 statusLine(ccstatusline、benabraham、ohugonnot…),天花板都是「只活在終端機」。
- 官方 issue #6577(放寬 statusLine context 資料)**Closed as not planned**。
- 「面板顯示用量」是已知、官方未做的洞(issue #33819 / #38791 / #21781);面板原生只有頂部一條**無數字**細進度條。
- **關鍵**:面板要顯示用量,社群**無人用 hook 塑進對話**——一律另裝 **companion VS Code 擴充**(Claude Context Bar / Token View / Prism),讀 transcript、印在 VS Code 自己的狀態列。

**決策(甲方拍板,2026-06-16)**:
- **面板呈現面 = 外部現成擴充 `Claude Context Bar`(`ezoosk.claude-context-bar`)**。讀同一份 `~/.claude/projects/` JSONL、只在本地算、MIT;預設 `warningThreshold 50%` = 200k 視窗的 100k,**黃燈正好壓在本工單的軟線上**,近乎零設定。**已安裝、實機顯示 31% 確認可用。**
  - 殘留風險(已對甲方講明、甲方接受):該擴充**源碼不可審**(Marketplace 有、找不到公開 repo),且讀**跨全專案**的 transcript;甲方以 Marketplace 信任換便利。
- **終端機呈現面 = 保留 `statusline/context-gauge.sh`** 當備援(在終端機 `claude` 仍正常運作、已驗)。雙軌。
- **「hook 印進對話」路線:查證後放棄**。理由:無社群先例(未踏之地)、`systemMessage` 面板渲染未知、每回合冒訊息恐成 nagware。既有 companion 擴充已零建置解決面板可見性。

**對交付物的影響**:D1/D2/D3 不變(statusLine 仍交付,定位改為終端機備援)。面板可見性由外部擴充承擔,**不再自建 hook 呈現層**。「加分項」之 model/% 顯示由 Claude Context Bar 的 tooltip 涵蓋。

## 一、背景與目標

這套 `dev-env-v3`（RELIABLE_WORKFLOW v3 harness：plan-gate / handoff / session-start 注入 / context gauge）目前只活在這台 Mac 的本機 repo。甲方要：

1. **備份**到自己的私有 GitHub repo（「任何地方都能拉下來、隨時更新」）。→ **Slice 0，已完成**。
2. 做一支**跨平台一鍵部署器**，讓在新機器（Windows / 其他 Mac）上能一鍵把這套環境裝起來。

「部署開發環境」有兩種語意，甲方拍板**兩者都做**，且在安裝過程**以互動選單讓使用者選**：

- **模式 A — 機器層安裝**：clone 本 repo + 把全域 statusLine 接進 `~/.claude/settings.json`（指向本機 clone 路徑的 `context-gauge.sh`）。每台機器一次。
- **模式 B — 專案層採用**：把 `hooks/` + `templates/policy.json` 複製進某個目標專案、並 merge 該專案的 `.claude/settings.json`（三個 hook 用 `$(git rev-parse --show-toplevel)` 自解析）。每個目標專案一次。

## 三、設計原則（不可違背）

1. **冪等（idempotent）**：安裝器可重複跑。重跑不得重複注入 hook、不得洗掉使用者既有設定。更新流程 = `git pull` + 重跑安裝器。（甲方核心需求：「任何地方隨時更新」。）
2. **依賴：一定 check，安裝採「徵同意」非靜默**。第一步跑 preflight，印 present/missing/版本分層報告。缺必要依賴 → 偵測套件管理器、**明示即將執行的指令、問 y/n** 再裝；偵測不到或缺的是 git 本身 → 印官方下載連結並停。缺可選依賴（jq）→ 問一下但不裝也照跑。提供 `--yes`/`--auto` 旗標跳過所有確認（真正無人值守一鍵）。**不靜默 escalate 權限。**
3. **非侵入 + 可一鍵回滾**：模式 A 動 user-level 前先備份 `~/.claude/settings.json`；只動/新增 `statusLine` 一鍵，保留其餘鍵。模式 B 只動目標 repo 內 `.claude/`，不碰其 user-level。回滾手段寫進 README。
4. **可攜 + 純 bash 3.2**：核心 merge / 路徑解析 / 依賴偵測只在 `install.sh` 實作一次；Windows 入口 `install.ps1` 為薄殼（檢查 Git Bash → 轉呼 install.sh）。腳本內零 LLM、零網路（除 `git clone`/`pull`）。
5. **Windows = 要求 Git Bash**（甲方拍板）：不重寫 hook 為 PowerShell/Node。Windows 上沿用同一套 bash 腳本，依賴 Git for Windows 提供 bash。

## 四、交付項目

- **D1. `install.sh`**（Mac / Linux / Git-Bash 共用，核心邏輯所在）
  - preflight 依賴檢查 + 分層報告 + 徵同意安裝（`--auto` 跳過）。
  - 模式 A：merge statusLine 進 `~/.claude/settings.json`（算本機 clone 絕對路徑、備份、保留既有鍵、冪等）。
  - 模式 B：`--adopt <repo>` 複製 hooks + templates、merge 目標 repo `.claude/settings.json`（冪等）。
  - 互動主選單：無旗標跑起來時問使用者 A / B / 兩者；旗標可非互動指定。
  - `--update`：`git pull` 後自動重套（冪等性保證更新安全）。
- **D2. `install.ps1`**（Windows 入口薄殼）：偵測 Git Bash（`bash`/Git for Windows），缺則印安裝引導；存在則以 bash 轉呼 `install.sh` 並轉傳旗標。
- **D3. README / bootstrap 章節**：跨平台 quickstart（`git clone <ssh-url> && ./install.sh`；Windows `install.ps1`）、互動選單說明、`--auto`/`--adopt`/`--update` 用法、一鍵回滾。

> JSON merge 策略：優先 `jq`（present 已確認）做結構化 merge；無 jq 時的 fallback 由工程師在計畫書提案（窄目標、保留既有鍵；若 fallback 風險過高，可將「無 jq 時 merge」降級為「印出待手動貼上的片段 + 明確指示」，由老師批板）。

## 一、背景與目標

`resume.md`（v3-005 handoff）解決「這個 session 改了什麼」——純 git 衍生、每次重生、**暫態**。它不記**為什麼**。AI 跨 session 最大的敵人不是忘記寫過的碼，而是 **Decision Drift**：

> 三週後冷啟動，Claude 恢復成「系統用 JWT」（肯定面留著），但漏掉「**不用 Refresh Token、以及為什麼**」（否定面掉了），於是自信地重新提案一個當初被否決的方案。

**Claude 從不忘記「我們用了 X」，它忘的永遠是「我們拒絕了 Y，以及為什麼」。** Decision Ledger 就是把這個否定面變成**被記錄的、可驗證的、append-only** 的耐久記憶。

**v3-008 整體願景**（分多票）：evidence-bound decision → constraint surfacing → architecture-review 路由。**本票（Slice 1）只做地基的地基：Type A**——證明「冷啟動能正確把過去的重要否定面帶回來」。這件事做不到，後面全是空中樓閣。

## 三、設計原則（不可違背）

1. **Append-only / immutable**：一筆決策入帳後**永不修改原檔**。推翻 = 寫一筆新決策 `supersedes: [舊id]`。理由：AI 很愛改歷史，但工程決策的價值就在「當初為什麼那樣想」，不是「現在覺得什麼合理」。
2. **否定面是一等公民**：每筆決策必須記 `rejected`（被拒方案 + why）。這是治 drift 的解藥，不是裝飾。沒有 why 的裸約束，三週後你自己也會問「為什麼不行？」然後重新推理。
3. **superseded_by 不物化**：被取代關係只存單向（新決策 `supersedes` 指向舊的）。`superseded_by` **不寫進舊檔**（寫了就違反 append-only）。Resume Reader **讀取時反查動態算**。決策數 < 50，全掃是 microseconds。
4. **falsifiable**：`evidence`（commit hash）+ `verification`（**冪等可重跑**命令）綁定。對不上 = invalid。可重跑的歸 `verification`，一次性觀測/感想歸 `note` 自由文字，**中間不留「長得像驗證、其實不能重跑」的灰色地帶**。
5. **Human Gate**：決策入帳要人點頭。`ledger-add` **拒絕沒有 approval reason 的入帳**（gate 是結構性的，不是禮貌性的）。Claude 起草 YAML、人 `y` + 一句理由才落盤。
6. **Resume 輸出物 = 否定面 + open questions，不是 decision-list dump、不是 confidence score**。score 是假精度（LLM 生不出校準信心值），拿掉；留可驗證的 open questions。
7. **純 bash 3.2、零 LLM、零網路**（除 git）。

## 四、Schema（鎖死，≤10 欄）

```yaml
id: AUTH-001              # 穩定識別碼，append-only 的主鍵
date: 2026-06-18
status: active            # active | superseded（顯式欄位是給人快速掃的冗餘；真值由 Reader 反查推導）
claim: JWT only           # 一行決策（肯定面）
rejected:                 # 否定面 —— drift 的真正解藥
  - option: JWT + Refresh Token
    why: 增加複雜度，系統僅內部使用
supersedes: []            # 只指向更舊的 id（append-only 不違反）；superseded_by 不存，Reader 算
evidence:
  commits: [a31f8f2]
  files: [auth/service.ts]   # optional
verification:             # 必須冪等可重跑；對不上 = invalid
  - npm test auth/*
open_questions: []        # optional；Resume 會聚合所有 active 決策的未解問題
note: |                   # 自由文字：不可驗證的觀測 + 入帳理由
  入帳理由：影響未來所有 auth 設計
```
